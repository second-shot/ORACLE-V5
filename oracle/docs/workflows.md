# Oracle Workflows

## Purpose
Workflows define how work moves through Oracle. They are not suggestions. They are the routes. Mia follows them. Humans follow them. Deviation is either a documented exception or a mistake.

---

## The Mission Loop

```
OBSERVE → MAP → DECIDE OR ESCALATE → EXECUTE → RECORD
```

### Observe
Read Oracle docs in navigation order before acting:
`vision.md` → `architecture.md` → `standards.md` → `backlog.md` → relevant mission → relevant instruction.

### Map
Before doing anything, map the terrain:
- What is the mission?
- What is already known?
- What is unknown or unclear?
- What decisions need to be made before acting?
- What is outside Mia's authority?

### Decide or Escalate
If the path is clear and within authority: act.
If the path requires a decision outside Mia's authority, or if critical information is missing: escalate before acting.

### Execute
Perform the work with a clear output and definition of done.

### Record
Every action taken, every decision made, every finding surfaced is recorded. Not summarised later — recorded as it happens.

## The Memory Workflow

Triggered when a mission surfaces a durable lesson, a system-level change, or a correction to an existing belief.

```
SURFACE → EVALUATE DURABILITY → WRITE TO MEMORY → LOG DATE AND SOURCE
```

Ask: "Will this matter in three months?"
- Yes → write to `oracle/docs/memory.md`
- No → leave it in the mission record only

---

## The Review Workflow

Triggered at the end of every mission before completion is declared.

```
RUN CHECKLIST → SURFACE GAPS → RESOLVE OR FLAG → CONFIRM COMPLETION
```

Full checklist: `oracle/mia/checklists/mission-review.md`

No mission is complete without a passed checklist.

---

## The Backlog Workflow

Triggered when a new task is identified, a task is deprioritised, or work is blocked.

```
IDENTIFY ITEM → ASSESS URGENCY AND IMPACT → ASSIGN STATUS → ADD TO BACKLOG
```

Statuses: `backlog` / `ready` / `active` / `blocked` / `review` / `done` / `archived`

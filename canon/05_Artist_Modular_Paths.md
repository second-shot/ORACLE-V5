# Artist Modular Paths

## Purpose
Define how ORACLE V5 adapts to different creative identities without changing the core engine.

## Rule
One core system.
Different domain layers.
No separate products.

## Shared core for every artist
- identity profile
- work archive
- project builder
- pricing + market routing
- proposal generation
- bounded agents
- archive resurfacing
- commercial split logic

## Painter path
- artwork cataloguing
- series clustering
- medium / dimensions / year
- pricing ladder
- collector / exhibition proposals
- archive by era, palette, motif

## Singer / musician path
- song archive
- release grouping
- sonic identity map
- rollout plans
- visual concept links
- sync / licensing / release proposals

## Filmmaker / director path
- treatments
- reference mapping
- body-of-work archive
- pitch decks
- campaign / client packaging

## Fashion / stylist path
- look archive
- garment / styling system
- editorial concepts
- rental / resale / client packaging
- aesthetic lineage

## Designer / visual artist path
- asset classification
- project grouping
- edition / product / commission paths
- proposal outputs

## Hybrid artist path
- combine modules without breaking the core object model
- allow one artist to hold multiple mediums and routes

## UI implication
Each artist sees:
- same core structure
- different labels
- different priority panels
- different default routes
- different proposal outputs

## Constraint
The modular path changes the workflow layer, not the soul of the system.
